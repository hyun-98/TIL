# 📚 TIL - 다중 서버 환경 구성 (Auto Scaling & Load Balancing)

## 📌 3장 핵심 개념 정리

이번 학습에서는 AWS 환경에서 **트래픽 증가와 장애 상황에 대응하는 구조**를 다뤘다.
핵심은 👉 **“자동 확장 + 트래픽 분산 + 장애 대응”**

---

## 1️⃣ AWS Auto Scaling 그룹

### ✔️ 개념

**Auto Scaling 그룹**은 동일한 환경의 EC2 인스턴스를 자동으로 생성/삭제하는 서비스다.

* 같은 AMI 기반으로 생성된 인스턴스 묶음
* 동일한 사양, 코드, 설정 유지
* 트래픽 변화에 따라 서버 수 자동 조절

### ✔️ 동작 방식

* 📈 사용자 증가 → 인스턴스 자동 증가 (Scale Out)
* 📉 사용자 감소 → 인스턴스 자동 감소 (Scale In)

### ✔️ 추가 기능

* ⏰ 시간 기반 스케일링

  * 특정 시간에 트래픽 증가 예상 시 미리 확장
  * 예: 이벤트 시작 전 서버 미리 확보

### 💡 왜 중요한가?

* 서버 과부하 방지
* 비용 최적화 (필요할 때만 서버 사용)
* 수동 대응 없이 자동 운영 가능

## 2️⃣ Elastic Load Balancing (ELB)

### ✔️ 개념

**ELB(Elastic Load Balancer)**는 클라이언트 요청을 여러 서버로 분산하는 서비스다.

### ✔️ 역할

* 사용자 요청을 먼저 받음
* 여러 EC2 인스턴스에 트래픽 분산
* 특정 서버에 부하 집중 방지

### ✔️ 핵심 기능

* ⚖️ 트래픽 분산 (Load Balancing)
* ❤️ Health Check

  * 정상 서버에만 요청 전달
  * 장애 서버는 자동 제외

### 💡 ELB가 없다면?

* L4 스위치 같은 물리 장비 직접 운영 필요
* 비용 증가 + 관리 복잡도 상승

### 🔥 핵심 포인트

👉 **“ELB = 서버 앞단의 교통 정리 담당”**

---

## 3️⃣ 장애 조치 아키텍처 (Failover)

### ✔️ 개념

**장애 조치(Failover)**는 일부 서버에 문제가 발생했을 때
정상 서버가 자동으로 트래픽을 대신 처리하는 구조다.

### ✔️ 동작 흐름

1. 서버 장애 발생
2. ELB가 Health Check로 감지
3. 해당 서버를 트래픽 대상에서 제외
4. 정상 서버로 요청 전달

### 💡 왜 중요한가?

* 서비스 중단 방지
* 사용자 경험 유지
* 장애 대응 자동화

---

## 🧠 전체 구조 흐름

```
사용자 → Load Balancer → Auto Scaling 그룹 (EC2 여러 대)
                          ↓
                    장애 발생 시 자동 제외
                          ↓
                    정상 서버로 트래픽 전달
```

---

## ✨ 오늘의 인사이트

* 단일 서버 구조는 장애에 매우 취약하다
* **확장성(Scalability)**과 **고가용성(High Availability)**은 필수
* AWS는 이를 자동화된 서비스로 제공한다는 점이 핵심
