# 🔐 AWS에서 Root 대신 IAM 사용이 중요한 이유

AWS에서 **Root 계정 대신 IAM(Identity and Access Management) 사용자 사용**은  
보안의 **가장 기본이자 핵심 원칙**입니다.

> 💡 비유: 집 전체 마스터키를 들고 다니기보다,  
> 용도에 맞는 열쇠를 따로 만들어 사용하는 것과 같습니다.

---

## 1️⃣ 권한 최소화 원칙 (Least Privilege)

Root 계정은 AWS의 모든 리소스와 결제 정보에 **무제한 접근 권한**을 가집니다.

### ✅ IAM 사용 시
- 특정 사용자에게 필요한 권한만 부여 가능
  - 예:  
    - S3 읽기 전용  
    - EC2 관리 권한  

### 🎯 장점
- 실수 또는 악의적인 행동으로 인한
  - 전체 시스템 삭제
  - 데이터 유출  
  👉 **사전 방지 가능**

---

## 2️⃣ 사고 발생 시 피해 최소화 (Blast Radius Reduction)

Root 계정이 유출되면:
- 계정 삭제
- 대규모 리소스 생성 (비용 폭탄)
👉 치명적인 피해 발생

### ✅ IAM 사용 시
- 유출된 계정의 **권한 범위 내에서만 피해 발생**

### 🚨 대응 방법
- 해당 IAM 사용자 즉시:
  - 비활성화
  - 삭제
  - 권한 회수

👉 **빠른 차단 가능**

---

## 3️⃣ 개별 식별 및 책임 추적 (Accountability)

Root 계정을 여러 명이 공유하면:
- 누가 무엇을 했는지 추적 불가능 ❌

### ✅ IAM 사용 시
- 사용자별 고유 ID 부여
- AWS CloudTrail로 활동 추적 가능

### 🎯 장점
- 보안 감사
- 장애 원인 분석
👉 **정확한 추적 가능**

---

## 4️⃣ 유연한 보안 정책 적용

IAM은 매우 세밀한 보안 설정이 가능합니다.

### 🔐 주요 기능
- 조건부 접근 제어
  - 회사 IP에서만 접속 허용
  - MFA 인증 시에만 권한 활성화
- 비밀번호 정책 설정
  - 주기적 변경
  - 복잡도 요구

👉 **기업 환경에 맞는 보안 설계 가능**

---

## 🚫 Root 계정은 언제 사용해야 할까?

Root 계정은 아래와 같은 **예외적인 경우에만 사용**해야 합니다:

- 초기 IAM 사용자 생성
- 결제 정보 변경
- 계정 해지
- 일부 Root 전용 작업

👉 그 외에는 **사용하지 않는 것이 원칙**

---

## 🔒 보안 Best Practice

- Root 계정에 **MFA(다중 인증)** 반드시 설정
- **Access Key 생성 금지 또는 삭제**
- 안전한 장소에 계정 정보 보관

---

## 📌 한 줄 정리

> ✅ IAM 사용자 사용 = 권한 최소화 + 사고 대응 + 추적 가능  
> ❌ Root 계정 상시 사용 = 보안 리스크 최대화
