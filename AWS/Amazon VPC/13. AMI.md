# AWS AMI (Amazon Machine Image) 정리

## 1. AMI란?

**AMI (Amazon Machine Image)** 는
Amazon Web Services에서 EC2 인스턴스를 생성할 때 사용하는 **서버 템플릿 이미지**입니다.

쉽게 말하면:

> “운영체제 + 소프트웨어 + 설정 상태를 그대로 복제한 스냅샷”

---

## 2. AMI 구성 요소

AMI는 다음 3가지로 구성됩니다:

### 2.1 루트 볼륨 (Root Volume)

* OS (Linux, Windows 등)
* 설치된 패키지 (nginx, httpd 등)
* 애플리케이션 코드

### 2.2 런치 권한 (Launch Permissions)

* AMI를 누가 사용할 수 있는지 설정

  * Private (나만)
  * Public (전체 공개)
  * 특정 계정 공유

### 2.3 블록 디바이스 매핑 (Block Device Mapping)

* EBS 볼륨 구성 정보
* 디스크 크기, 타입 등 정의

---

## 3. AMI 종류

### 3.1 Public AMI

* AWS 또는 커뮤니티에서 제공
* 예: Amazon Linux, Ubuntu

### 3.2 Private AMI

* 사용자가 직접 생성
* 조직 내부에서만 사용

### 3.3 Marketplace AMI

* AWS Marketplace에서 제공
* 유료/무료 소프트웨어 포함

---

## 4. AMI 생성 과정

### 4.1 기본 흐름

1. EC2 인스턴스 생성
2. 필요한 설정 적용 (패키지 설치, 설정 변경)
3. AMI 생성 (Create Image)

---

### 4.2 CLI 예시

```bash
aws ec2 create-image \
  --instance-id i-1234567890abcdef0 \
  --name "my-ami" \
  --no-reboot
```

---

## 5. AMI 기반 인프라 운영 방식

### 5.1 Immutable Infrastructure

> 서버를 수정하지 않고, 새로운 AMI로 교체

#### 특징

* 기존 서버 변경 ❌
* 새 AMI 생성 후 교체 ⭕

#### 장점

* 일관성 유지
* 롤백 쉬움
* 장애 대응 빠름

---

### 5.2 Auto Scaling과의 관계

AMI는 다음과 함께 사용됨:

* Launch Template
* Auto Scaling Group

👉 새로운 인스턴스는 항상 동일한 AMI로 생성됨

---

## 6. AMI 사용 시 장점

* 빠른 서버 생성
* 동일한 환경 복제 가능
* 배포 자동화에 적합
* 장애 시 빠른 복구

---

## 7. AMI 관리 전략 (실무 중요)

### 7.1 버전 관리

* 예:

  * app-ami-v1
  * app-ami-v2

### 7.2 최신 AMI 유지

* 보안 패치 포함해서 주기적으로 갱신

### 7.3 불필요 AMI 삭제

* 오래된 AMI → 비용 증가 원인

---

## 8. AMI vs Docker 이미지

| 항목    | AMI   | Docker |
| ----- | ----- | ------ |
| 범위    | OS 전체 | 애플리케이션 |
| 부팅 속도 | 느림    | 빠름     |
| 사용 위치 | EC2   | 컨테이너   |
| 관리 단위 | VM    | 컨테이너   |

---

## 9. 주의사항

* AMI 생성 시 디스크 크기 확인
* 민감 정보 포함 여부 확인 (SSH 키, 비밀번호 등)
* 리전 간 공유 시 복사 필요

---

## 10. 한 줄 정리

> AMI는 “서버 상태를 그대로 복제해서 재사용하는 이미지 템플릿”이다.
