# 📚 TIL - 5장 배포 과정 (Deployment Strategies)

## 📌 핵심 요약

배포는 단순히 코드를 서버에 올리는 것이 아니라
👉 **서비스 중단 없이 안정적으로 변경을 적용하는 과정**이다.

---

# 🚀 1. 무중단 배포 vs 중단 배포

## ✔️ 중단 배포 (Downtime Deployment)

* 서버를 **잠시 멈춘 후 배포**
* 배포 중에는 사용자 접근 불가

### 🔻 특징

* 구현이 단순함
* 장애 가능성 낮음 (구조 단순)
* ❌ 서비스 중단 발생

### 💡 사용 상황

* 내부 시스템
* 트래픽이 거의 없는 서비스
* 초기 개발 단계

---

## ✔️ 무중단 배포 (Zero-Downtime Deployment)

* 서비스를 **중단하지 않고 배포**
* 사용자 입장에서는 끊김 없음

### 🔻 특징

* 높은 사용자 경험(UX)
* 실서비스 환경 필수
* ❗ 구조가 상대적으로 복잡

---

# ⚙️ 2. 현재 위치 배포 (Rolling Deployment)

👉 가장 기본적인 무중단 배포 방식

## ✔️ 개념

* 여러 대의 서버를 **순차적으로 업데이트**
* 일부 서버만 교체하면서 서비스 유지

---

## 🔄 동작 방식

```text id="roll1"
[서버 1, 2, 3, 4 운영 중]

1️⃣ 서버 1 배포 → 트래픽 제외 → 업데이트 → 복귀
2️⃣ 서버 2 배포 → 동일 과정 반복
3️⃣ 서버 3, 4 순차 진행
```

---

## 👍 장점

* 추가 서버 필요 없음 (비용 효율적)
* 구현 난이도 낮음
* 점진적 배포 가능

---

## 👎 단점

* 배포 중 **버전이 섞임**
* 호환성 문제 발생 가능
* 롤백이 까다로움

---

## 💡 실무 포인트

* 로드 밸런서(ALB/Nginx)에서
  👉 **트래픽을 제거 → 배포 → 복귀** 방식으로 처리

---

# 🔵🟢 3. 블루/그린 배포 (Blue-Green Deployment)

👉 가장 안정적인 무중단 배포 방식 중 하나

---

## ✔️ 개념

* 동일한 환경을 **두 개(Blue / Green)** 운영
* 한쪽은 운영, 한쪽은 새 버전 배포

---

## 🔄 동작 방식

```text id="bg1"
[Blue = 운영 중]
[Green = 새 버전 배포]

1️⃣ Green 환경에 새 버전 배포
2️⃣ 테스트 진행
3️⃣ 트래픽을 Blue → Green으로 전환
```

---

## 👍 장점

* 롤백이 매우 빠름 (트래픽만 되돌리면 됨)
* 안정성이 매우 높음
* 테스트 환경 확보 가능

---

## 👎 단점

* 서버가 2배 필요 (비용 증가)
* 인프라 관리 복잡

---

## 💡 실무 포인트

* AWS에서는:

  * ALB Target Group 전환
  * Route 53 가중치 기반 라우팅

---

# ⚖️ 배포 전략 비교

| 구분     | 중단 배포 | 롤링 배포 | 블루/그린 배포 |
| ------ | ----- | ----- | -------- |
| 서비스 중단 | 있음 ❌  | 없음 ✅  | 없음 ✅     |
| 안정성    | 보통    | 보통    | 높음 ⭐     |
| 비용     | 낮음    | 낮음    | 높음 ❗     |
| 롤백     | 어려움   | 어려움   | 매우 쉬움    |
| 운영 난이도 | 낮음    | 중간    | 높음       |

---

# 🧠 핵심 이해 포인트

## ✔️ 왜 무중단 배포가 중요한가?

* 서비스 신뢰도 유지
* 사용자 이탈 방지
* 장애 최소화

---

## ✔️ 언제 어떤 방식을 선택할까?

* 초기 서비스 👉 중단 배포
* 일반 서비스 👉 롤링 배포
* 대규모 서비스 👉 블루/그린 배포

---

# 💡 추가 개념 (알아두면 좋은 것)

## 🟡 카나리 배포 (Canary Deployment)

* 일부 사용자에게만 새 버전 적용
* 문제 없으면 전체 확장

👉 블루/그린 + 롤링의 중간 형태
