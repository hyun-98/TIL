# 📚 TIL - 운영 서버의 외부 환경 구성 (DNS & HTTPS)

## 📌 4장 핵심 개념 정리

실제 서비스를 운영하려면 단순히 서버를 띄우는 것만으로는 부족하다.
👉 **도메인 연결 + 안전한 통신(HTTPS)**이 반드시 필요하다.

---

## 1️⃣ 도메인과 DNS

### ✔️ 도메인이란?

* 사람이 기억하기 쉬운 주소
* 예: `www.naver.com`

👉 실제로는 IP 주소를 대신하는 이름이다.

---

### ✔️ DNS (Domain Name System)

* 도메인 ↔ IP 주소를 연결해주는 시스템
* 인터넷의 “전화번호부” 역할

### ✔️ 동작 과정

1. 사용자가 도메인 입력
2. DNS 서버에 요청
3. 해당 도메인의 IP 주소 반환
4. 해당 서버로 접속

---

### ✔️ AWS Route 53

* 도메인 구매 및 관리 서비스
* DNS 설정 및 트래픽 라우팅 가능
* 고가용성 DNS 제공

---

## 2️⃣ HTTPS와 SSL/TLS

### ✔️ HTTPS란?

* HTTP에 보안을 추가한 프로토콜
* SSL/TLS 기반으로 동작

👉 쉽게 말하면
👉 **“암호화된 HTTP”**

---

## 🔐 HTTPS의 3가지 핵심 목적

### 1. 기밀성 (Confidentiality)

* 데이터를 암호화하여 중간 탈취 방지
* 해커가 내용을 읽을 수 없음

---

### 2. 무결성 (Integrity)

* 데이터가 중간에 변조되지 않도록 보호
* 위변조 공격 방지

---

### 3. 인증 (Authentication)

* 접속한 서버가 진짜인지 확인
* 피싱 및 위장 서버 방지

---

## ⚠️ HTTPS는 선택이 아니라 필수

과거:

* 로그인 / 결제 페이지에만 사용

현재:

* **모든 페이지에 적용 권장**
* SEO 및 브라우저 보안 정책에도 영향

---

## 🔄 HTTP vs HTTPS

| 구분  | HTTP  | HTTPS  |
| --- | ----- | ------ |
| 보안  | 없음    | 암호화    |
| 데이터 | 평문 전송 | 암호화 전송 |
| 신뢰성 | 낮음    | 높음     |

---

## 🧠 HTTPS 동작 방식 (핵심 흐름)

```id="https-flow-diagram"
[1] 클라이언트 → 서버 접속 요청
        ↓
[2] 서버 → SSL 인증서 전달
        ↓
[3] 클라이언트 → 인증서 검증
        ↓
[4] 암호화 키(세션 키) 생성 및 교환
        ↓
[5] 이후 통신은 모두 암호화된 상태로 진행 🔐
```

## 🔐 HTTPS는 크게 3단계로 작동

① TCP 연결 — 먼저 일반 TCP 3-way 핸드셰이크로 연결을 수립해요 (SYN → SYN-ACK → ACK).
② TLS 핸드셰이크 — 본격적인 암호화 설정 단계예요.

클라이언트가 지원하는 암호화 알고리즘 목록을 보내고 (Client Hello)
서버가 알고리즘을 선택하고 공개키가 담긴 SSL 인증서를 보내줘요
브라우저는 인증서를 CA(인증기관)를 통해 신뢰할 수 있는지 검증하고
Pre-master Secret을 서버의 공개키로 암호화해서 전송해요
양쪽이 이 값을 기반으로 같은 세션 키(대칭키)를 독립적으로 생성해요

③ 암호화 통신 — 이후 모든 HTTP 요청/응답은 세션 키로 암호화돼서 주고받아요.

## 💡 쉽게 이해하기

* HTTP = 엽서 (누구나 내용 볼 수 있음)
* HTTPS = 봉투 + 자물쇠 (내용 보호됨)
* SSL/TLS 인증서는 HTTPS 통신을 위한 것이기 때문에 클라이언트와 직접 통신하는 서버에 설치돼 있어야 한다
* 📌 AWS Certificate Manager를 통한 SSL/TLS 인증서 등록
